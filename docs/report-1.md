# SimpleSurvey 项目完成报告

> 项目名称：SimpleSurvey 在线问卷系统  
> 完成时间：2026-04-02  
> 技术栈：Go + Gin、Vue 3、MongoDB、Redis、Docker Compose

## 1. 项目目标

本项目的目标是实现一个可本地一键部署、具备完整业务闭环的在线问卷系统，核心目标包括：

1. 支持用户注册/登录、问卷创建、发布、填写、统计、关闭的全流程。
2. 支持多题型（单选、多选、文本、数字）及题目级约束。
3. 支持基于条件的问卷内跳题（逻辑跳转）。
4. 支持匿名填写与权限分级（普通用户、管理员）。
5. 提供可维护的工程结构与可扩展的数据模型。

## 2. 系统设计

系统采用“前后端分离 + 模块化单体”架构，整体分层如下：

- **接入层**：前端使用 Vue 构建，Nginx 托管静态资源，并将 `/api/*` 代理到后端服务。
- **应用层**：Go + Gin，遵循 `delivery -> service -> repository` 分层，业务逻辑与存储实现解耦。
- **缓存层**：Redis 用于令牌/会话相关存储（如 refresh token）。
- **持久层**：MongoDB 存储用户、问卷、答卷等核心业务数据。

当前 `docker-compose.yml` 统一编排 `mongo`、`redis`、`backend`、`frontend` 四个服务，对外入口为 `http://localhost:3000`。

后端接口以 `/api/v1` 分组，并按业务划分为：

- `/auth/*`：注册、登录、刷新令牌
- `/questionnaires/*`：问卷创建、列表、状态更新、统计、答卷明细
- `/surveys/*`：填写端获取问卷、提交答卷
- `/admin/*`：管理员用户与问卷管理

## 3. MongoDB 设计

### 3.1 集合设计

系统主要使用 3 个集合：

1. `users`：用户账号、角色、状态、创建时间等
2. `questionnaires`：问卷元信息、题目结构、逻辑规则、状态、截止时间
3. `responses`：答卷内容、匿名标识、提交时间、统计字段

### 3.2 索引设计

结合仓储实现，已建立关键索引：

- `users`
	- `uq_users_username`（唯一索引）：保证用户名唯一
	- `idx_users_role_status_created`：管理员分页筛选场景优化
- `questionnaires`
	- `idx_questionnaires_owner_status_created`：创建者维度查询与状态筛选
	- `idx_questionnaires_is_deleted`：逻辑删除过滤
- `responses`
	- `idx_responses_qid_submitted`：按问卷时间倒序查询答卷
	- `idx_responses_answer_qid`：按题目筛选答案明细

### 3.3 数据一致性策略

- 领域层统一以字符串 ID 暴露，仓储层负责 `ObjectID <-> string` 转换。
- 问卷与答卷均记录 UTC 时间字段，保证排序与统计口径一致。
- 通过 `isDeleted` 支持逻辑删除，避免物理删除影响统计追溯。

## 4. 跳转设计

本项目的“跳转”包含两类：

### 4.1 页面路由跳转（Vue Router）

- 页面路由：`/login`、`/register`、`/`、`/create`、`/survey/:id`、`/stats/:id`
- 路由守卫策略：
	- 需要登录页面（`meta.auth`）无 token 时跳转 `/login`
	- 访客页面（`meta.guest`）已有 token 时跳转 `/`

### 4.2 问卷内题目跳转（逻辑规则）

填写页根据当前题作答结果实时计算下一题：

1. 读取当前题相关 `logicRules`
2. 按操作符匹配条件：`EQUALS` / `CONTAINS` / `GREATER_THAN` / `LESS_THAN`
3. 命中后执行 `JUMP_TO`，定位 `targetQuestionId`
4. 未命中则按默认顺序进入下一题

同时维护历史栈支持“上一题”回退，确保跳题场景下仍可回溯修改答案。

## 5. 校验设计

采用“前端交互校验 + 后端权威校验”双层策略。

### 5.1 前端校验

- 创建问卷时校验：标题非空、至少一题
- 填写问卷时按题型执行即时校验：
	- 单选：必须选择
	- 多选：最小/最大选项数量
	- 文本：最小/最大长度
	- 数字：数字合法性、整数约束、最小/最大值

### 5.2 后端校验

后端在服务层进行最终一致性校验，避免绕过前端：

- 问卷结构校验：
	- `questionId` 唯一、题目标题/选项合法
	- 多选/文本/数字约束边界合法（如最小值不大于最大值）
	- 跳转规则必须引用存在题目、动作仅支持 `JUMP_TO`
	- 按题型限制可用操作符（如单选仅 `EQUALS`）
- 答卷校验：
	- 禁止重复作答同一题
	- 必答题必须填写
	- 选项值必须落在题目定义集合内

## 6. AI 使用过程

本项目在开发与文档阶段引入 AI 辅助（GitHub Copilot，模型：GPT-5.3-Codex），主要使用方式如下：

1. **需求拆解与接口草案**：先将问卷业务拆为认证、问卷管理、填写、统计、管理端模块。
2. **代码辅助生成**：辅助生成部分样板代码（路由、请求封装、基础表单结构），再由人工逐段审查并改造。
3. **校验与跳转逻辑完善**：用于补充边界条件清单（如多选上下限、数字比较符）。
4. **文档整理**：结合代码现状生成并统一 API、架构、测试与本完成报告。

执行原则：**AI 负责提速，人类负责验收**，所有核心逻辑均以代码与实测结果为准。

## 8. 遇到的问题

* 前端路由守卫与后端权限校验的边界设计，初期过于依赖前端状态。
* 部署较为复杂，需要多个步骤
* 前期难以控制用户的生命周期，存在安全隐患

## 9. 如何解决

* 通过后端中间件强化权限校验，确保无论前端如何跳转都无法绕过认证。
* 引入 Docker Compose 统一编排，简化部署流程。
* 引入 Redis 存储 refresh token，支持安全的令牌刷新机制，减少长期有效 token 的风险。
