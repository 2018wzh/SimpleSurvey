1: 项目：基于 MongoDB 的问卷系统开发
2: 一、项目目标
3: 本项目要求开发一个简化版的在线问卷系统，功能类似于“问卷星”或 Google Forms。
4: 系统应允许注册用户创建问卷，并通过URL分发问卷，其他用户可以填写问卷，系统可以统计问卷结果。
5: 本项目重点考察：
6: MongoDB 数据库设计能力 
7: 后端程序设计能力 
8: 数据结构与业务逻辑建模能力 
9: 条件逻辑与跳转逻辑的实现能力 
10: 使用 AI 编程工具的能力与规范 
11: 本项目不要求复杂界面，可以使用简单网页。
12: 评分重点：
13: 数据库设计 + 后端逻辑 + 文档质量 + AI使用过程说明
14: 
15: 二、项目说明
16: 本课程项目将模拟真实互联网应用开发的过程。
在实际的软件开发中，用户需求往往不是一次性确定的，而是会随着使用过程不断增加和变化。开发者通常需要在已有系统的基础上进行修改和扩展，而不是每次都重新设计和实现整个系统。
17: 为了让大家体验这种真实的软件开发过程，本项目分为 两个阶段 完成。
18: 
19: 第一阶段：根据初始需求完成系统设计与实现
20: 在第一阶段会提供一份初始用户需求说明。
21: 同学需要根据这份需求：
22: 进行系统设计 
23: 完成数据库设计 
24: 实现前后端程序 
25: 完成必要的测试 
26: 编写项目文档 
27: 提交完整的项目报告 
28: 整个过程尽可能利用AI编程工具完成。
29: 第一阶段提交的内容参见“七、提交内容”。 
30: 第一阶段的成果将作为第二阶段的基础。
31: 
32: 第二阶段：根据用户需求变更对系统进行修改与扩展
33: 在第一阶段提交完成之后，教师会发布一份新的用户需求变更说明。
34: 该需求变更模拟真实应用中的情况，例如：
35: 用户提出新的功能要求 
36: 原有功能需要扩展 
37: 原有设计无法满足新的使用方式 
38: 数据需要以新的方式组织和统计 
39: 在第二阶段，同学需要：
40: 在第一阶段系统的基础上进行修改和扩展 
41: 根据新的需求调整数据库设计 
42: 修改程序实现 
43: 保证旧功能仍然可用（除非需求明确要求改变） 
44: 完成新的测试 
45: 再次提交完整的项目报告 
46: 整个过程也尽可能利用AI编程工具完成。
47: 
48: 重要要求：不得推倒重做
49: 第二阶段 不允许丢弃第一阶段的成果重新实现。
50: 必须满足以下要求：
51: 必须在第一阶段代码基础上修改 
52: 不得重新设计一个完全新的系统替代原系统 
53: 不得删除第一阶段的主要功能 
54: 必须保留第一阶段的核心结构，并进行合理扩展 
55: 在最终报告中必须说明：
56: 第一阶段的设计是什么 
57: 新需求带来了哪些问题 
58: 为什么需要修改原设计 
59: 修改了哪些部分 
60: 数据结构如何变化 
61: 程序如何调整 
62: 是否遇到兼容性问题 
63: 如何保证旧功能仍然可用 
64: 即需要清楚记录整个修改过程。
65: 
66: 为什么分成两个阶段
67: 本项目的目标不是实现一个功能最多的系统，而是让同学体验：
68: 用户需求不确定 
69: 需求会不断变化 
70: 数据结构需要调整 
71: 系统需要迭代升级 
72: 旧数据必须保留 
73: 旧功能不能随意删除 
74: 这些都是实际互联网应用开发中的常见情况。
75: 通过两个阶段的开发，希望同学能够理解：
76: 软件系统不是一次设计完成的，而是在不断变化的需求中逐步演化的。
77: 
78: 三、技术要求
79: 须满足：
80: 后端数据库：MongoDB 
81: 编程语言：不限（Python / Node.js / Java / Go 等均可） 
82: 后端框架：不限 
83: 前端：不限，可非常简单 
84: 可以使用 AI 编程工具（ChatGPT / Copilot / Claude / Cursor 等） 
85: 须满足：
86: 所有数据存储在 MongoDB 中 
87: 不允许使用关系数据库 
88: 不允许用现成问卷系统框架 
89: 
90: 四、MongoDB 设计要求（重点）
91: 建议：
92: 勿将所有数据存在一个集合 
93: 谨慎使用冗余设计 
94: 勿把逻辑写死在代码 
95: 须说明：
96: 为什么这样设计 
97: 为什么不用关系数据库结构 
98: 为什么适合 MongoDB

177: 初始用户需求
178: 一、用户账号功能
179: 系统需要支持用户注册和登录。
180: 用户希望能够：
181: 注册账号 
182: 登录账号 
183: 使用自己的账号创建问卷 
184: 查看自己创建的问卷 
185: 查看问卷的统计结果 
186: 每个用户至少需要有：
187: 用户名 
188: 密码 
189: 注册时间 
190: 用户登录后，应只能看到自己创建的问卷。
191: 
192: 二、问卷创建与管理
193: 用户希望能够创建问卷。
194: 创建问卷时可以填写：
195: 问卷标题 
196: 问卷说明 
197: 是否允许匿名填写 
198: 用户希望能够：
199: 查看自己创建的问卷列表 
200: 发布问卷 
201: 设置问卷截止时间
202: 关闭问卷 
203: 每个问卷都应该有一个可以访问的链接，例如：
204: /survey/xxxxxx
205: 用户可以把这个链接发给别人，让别人填写问卷。
206: 填写问卷的人需要先登录。
207: 
208: 三、问卷中的题目
209: 一个问卷可以包含多个题目。
210: 用户希望在问卷中添加不同类型的问题。
211: 系统至少需要支持以下三种题型。
212: 
213: 1. 单选题
214: 示例：
215: 你的性别
216: 男 
217: 女 
218: 其他 
219: 用户希望能够：
220: 设置选项 
221: 设置是否必须回答 
222: 如果是必答题，填写问卷时必须选择。
223: 
224: 2. 多选题
225: 示例：
226: 你喜欢的水果
227: 苹果 
228: 香蕉 
229: 西瓜 
230: 葡萄 
231: 用户希望可以限制选择数量，例如：
232: 至少选择 2 个 
233: 最多选择 3 个 
234: 必须选择 2 个 
235: 系统在填写时需要检查是否符合要求。
236: 
237: 3. 填空题
238: 填空题可以是文字或数字。
239: 用户希望可以创建：
240: 文本填空 
241: 数字填空 
242: 对于文本填空，希望可以限制：
243: 最少输入多少字 
244: 最多输入多少字 
245: 对于数字填空，希望可以限制：
246: 最小值 
247: 最大值 
248: 是否必须是整数 
249: 例如：
250:    年龄：必须为整数，且在0~120之间
251:    手机号：必须为整数
252: 填写时系统应检查是否合法。
253: 
254: 四、题目跳转功能
255: 用户希望问卷可以根据回答情况跳转到不同题目。
256: 例如：
257: 如果第1题选 A → 跳到第5题
如果第1题选 B → 跳到第3题
258: 用户希望可以设置：
259: 根据单选结果跳转 
260: 根据多选结果跳转 
261: 根据数字填空内容跳转 
262: 填写问卷时，系统应自动跳到对应题目。
263: 用户不希望跳转写死在程序里，而是可以在问卷中设置。
264: 
265: 五、填写问卷
266: 填写问卷的人可以通过问卷链接进入问卷。
267: 填写时希望：
268: 按顺序回答问题 
269: 根据规则跳转 
270: 如果填写不符合要求，系统提示错误 
271: 填写完成后可以提交 
272: 每次填写都应该被保存。
273: 系统需要支持：
274: 多人同时填写 
275: 匿名填写 
276: 同一个人填写多次（如果允许） 
277: 
278: 六、查看统计结果
279: 创建问卷的人希望可以查看统计结果。
280: 统计结果至少包括：
281: 单选题
282: 每个选项有多少人选择 
283: 总共有多少人回答 
284: 多选题
285: 每个选项被选择多少次 
286: 填空题
287: 可以查看所有填写内容 
288: 如果是数字，可以查看平均值 
289: 用户希望能够：
290: 查看整个问卷的统计 
291: 查看某一道题的统计 
292: 统计结果应该根据已经填写的数据自动生成。
